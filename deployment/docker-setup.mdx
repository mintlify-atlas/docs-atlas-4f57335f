## Overview

Barro Tal Vez uses Docker Compose to orchestrate three containerized services:

- **PostgreSQL 15** - Database service (port 5433)
- **Express API** - Node.js backend (port 3001)
- **React Frontend** - Nginx-served SPA (port 8082)

All services are configured with automatic restarts and proper dependency management.

## Service Architecture

### Database Service (db)
- **Image**: `postgres:15`
- **Container**: `barro-db`
- **Port**: `5433:5432` (host:container)
- **Volume**: `postgres_data` for data persistence

### Backend Service (backend)
- **Build**: `./backend` directory
- **Container**: `barro-api`
- **Port**: `3001:3000`
- **Dependencies**: Requires `db` service
- **Volume**: `./backend/uploads` for persistent file storage

### Frontend Service (frontend)
- **Build**: `./frontend` directory
- **Container**: `barro-web`
- **Port**: `8082:80`
- **Dependencies**: Requires `backend` service
- **Web Server**: Nginx with custom configuration

- Docker Compose 2.0+

## Deployment

<Steps>
  <Step title="Clone the repository">
    Navigate to your project directory containing the `docker-compose.yml` file.

    ```bash
    cd /path/to/barro-tal-vez
    ```
  </Step>

  <Step title="Build the services">
    Build all Docker images defined in the compose file:

    ```bash
    docker compose build
    ```

    This will:
    - Build the backend image with Node.js 20 Alpine
    - Generate Prisma client
    - Build the frontend with Vite and package with Nginx
  </Step>

  <Step title="Start all services">
    Launch all services in detached mode:

    ```bash
    docker compose up -d
    ```

    Docker Compose will:
    1. Start the PostgreSQL database
    2. Wait for the database, then start the backend
    3. Wait for the backend, then start the frontend
  </Step>

  <Step title="Run database migrations">
    Apply Prisma migrations to set up the database schema:

    ```bash
    docker compose exec backend npx prisma migrate deploy
    ```
  </Step>

  <Step title="Seed the database (optional)">
    Populate the database with initial test data:

    ```bash
    docker compose exec backend npx prisma db seed
    ```

    This creates a test user:
    - Email: `test@barro.com`
    - Password: `123`
    - Role: `alumna`
  </Step>

  <Step title="Verify deployment">
    Check that all services are running:

    ```bash
    docker compose ps
    ```

    Access the application:
    - Frontend: http://localhost:8082
    - Backend API: http://localhost:3001
    - PostgreSQL: localhost:5433
  </Step>
</Steps>

## Volume Configuration

### Persistent Storage

The application uses two types of volumes:

1. **Named Volume** (`postgres_data`)
   - Stores PostgreSQL data
   - Persists across container restarts
   - Located in Docker's volume directory

2. **Bind Mount** (`./backend/uploads`)
   - Stores uploaded photos
   - Maps to host filesystem
   - Allows direct access to uploaded files

### Managing Volumes

<CodeGroup>
```bash List volumes
docker volume ls
```

```bash Inspect volume
docker volume inspect barro-tal-vez_postgres_data
```

```bash Remove volume (⚠️ deletes data)
docker volume rm barro-tal-vez_postgres_data
```
</CodeGroup>

## Troubleshooting

### Port Already in Use

**Error**: `Bind for 0.0.0.0:5433 failed: port is already allocated`

**Solution**: Change the host port in `docker-compose.yml`:

```yaml
ports:
  - "5434:5432"  # Changed from 5433 to 5434
```

### Backend Can't Connect to Database

**Error**: `Can't reach database server at db:5432`

**Solution**: Ensure the database is healthy:

```bash
docker compose logs db
docker compose exec db pg_isready -U postgres
```

### Permission Denied on Uploads Directory

**Error**: `EACCES: permission denied, mkdir '/app/uploads'`

**Solution**: The backend Dockerfile already handles this with:

```dockerfile
RUN mkdir -p uploads && chown -R node:node /app
USER node
```

If issues persist, check host directory permissions:

```bash
chmod 755 ./backend/uploads
```

### Prisma Client Not Generated

**Error**: `Cannot find module '@prisma/client'`

**Solution**: Regenerate Prisma client:

```bash
docker compose exec backend npx prisma generate
docker compose restart backend
```

### Container Keeps Restarting

**Solution**: Check container logs for errors:

```bash
docker compose logs backend --tail=50
```

Common causes:
- Database not ready (wait longer or add healthcheck)
- Missing environment variables
- Application crashes on startup

### Database Migration Fails

**Solution**: Check migration status and reset if needed:

```bash
# Check migration status
docker compose exec backend npx prisma migrate status

# Reset database (⚠️ deletes all data)
docker compose exec backend npx prisma migrate reset
```

## Production Considerations

<Warning>
  The default `docker-compose.yml` contains development credentials. **Never use these in production!**
  
  Update these values before deploying:
  - `POSTGRES_PASSWORD`
  - `JWT_SECRET`
  - Database connection string in `DATABASE_URL`
</Warning>

### Security Hardening

1. **Use Docker secrets** for sensitive data
2. **Enable TLS** for PostgreSQL connections
3. **Run services as non-root** (already configured for backend)
4. **Use specific image tags** instead of `latest`
5. **Limit container resources** with CPU and memory constraints

### Health Checks

Add health checks to `docker-compose.yml`:

```yaml
db:
  healthcheck:
    test: ["CMD-SHELL", "pg_isready -U postgres"]
    interval: 10s
    timeout: 5s
    retries: 5

backend:
  healthcheck:
    test: ["CMD", "curl", "-f", "http://localhost:3000/health"]
    interval: 30s
    timeout: 10s
    retries: 3
```
